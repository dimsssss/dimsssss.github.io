## 가
회사에서 팀 전체가 여태껏 개발했던 API를 테스트해보기로 하였다. 내 차례가 되어서 몇 가지를 테스트해보았는데 에러가 많이 나왔다. 그 이유는 관련된 테이블이 계속 수정되거나 RestAPI가 계속 바뀌는데 적용을 안해서 발생한 문제였다.

동작하지 않는 API는 사용하지 않는 API였고 제대로 된 기획이 나온 것들이 아니어서 결국엔 바뀔 수 가 있다. 그러다 보니 기획에 나온 것만(이것 도 사용을 안 할 수 있다고 한다) 수정하고 신경을 쓰다 보니 그 외에 API는 관리를 하지 않았다

어찌 되었건 결과적으로 동작을 안 하는 상황이니 고친다는 것은 당연한데 내가 문제라고 생각하는 것은 수정할 때마다 확인을 해보는 게 너무 귀찮고 시간이 많이 걸린다. 이에 대한 해결책으로 테스트 코드를 생각하고 있다
## 나
현재 팀에서 테스트 코드를 작성하고는 있는데 아니 정확하게 말하자면 테스트 코드를 도입을 하였지만 테스트 코드보다는 구현이 우선, 테스트 코드를 작성할 시간이 없다면 작성 안 해도 된다. 라는 시니어의 지침과 다들 귀찮아하는 분위기 때문에 유명무실해지고 있었다.
## 다
테스트의 종류에도 여러가지가 있는데 어떤 테스트 코드를 작성 해야 할까? 단위테스트, 통합테스트, e2e 테스트가 있는데 회사에서는  Mock을 곁들인 e2e 테스트 코드를 작성하고 있다.
데이터베이스 같은 외부 시스템 코드에 Mock을 사용하고 그 외는 supertest 모듈을 이용해  request 요청을 직접 발생한다.
## 라
내 코드에는 복잡한 로직이나 계산이 없고 데이터베이스에 접근하는 CRUD 코드가 전부이다.  전체 API 테스트 때 발생한 코드에 에러 내용도 데이터베이스의 변경 사항을 적용하지 않아서 발생하는 문제인데(컬럼의 수정 사항) 직접 테스트 하지 않고 Mock으로 처리하면 내 문제를 해결할 수 있나?
## 마
데이터베이스에 직접 쿼리를 날려봐야 확인을 할 수 있는 내용인데 그 부분을 Stub으로 대체하면 테스트를 하나 마나라고 생각한다. 팀에서는 데이터베이스 관련된 코드는 Mock을 쓰라고 했는데 내 상황을 이야기하고 Mock을 사용하지 않고 테스트 코드를 작성하는 것에 대해 이야기를 해봐야 할 것 같다.
